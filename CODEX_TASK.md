# Codex 작업 지시서 — 팔도상단: 조선의 바람

## 역할

당신은 이 저장소를 기반으로 **실행 가능한 모바일 웹앱 게임 프로토타입**을 구현하는 소프트웨어 엔지니어다.

## 최종 목표

`starter/` 폴더를 실제 Vite + React + TypeScript 앱으로 완성한다. 앱은 모바일 세로 화면에서 동작해야 하며, 사용자는 조선 후기 상인/선장으로 시작해 항구를 이동하고 상품을 사고팔며 돈을 벌 수 있어야 한다.

## 공식 Codex use case 반영 작업 방식

이 패키지는 Codex의 브라우저 게임 제작 흐름과 작은 UI 수정 흐름을 반영한다.

- 구현 전 `PLAN.md`를 확인한다.
- 작업 규칙은 `AGENTS.md`를 따른다.
- 가능한 경우 live browser 또는 Playwright로 실제 앱을 확인한다.
- 큰 작업은 milestone 순서대로 진행한다.
- UI polish는 한 번에 하나의 작은 변경으로 진행한다.
- 작업 로그는 `.logs/`에 남긴다.
- 향후 이미지 생성 프롬프트는 `.prompts/`에 저장한다.

## 반드시 구현할 기능

### 1. 데이터 로딩

- `public/data/*.json` 파일을 앱에서 로드한다.
- 항구, 상품, 배, 수레, 루트, 이벤트, 퀘스트, 월별 이벤트는 코드에 대량 하드코딩하지 않는다.
- TypeScript 타입을 정의하고 데이터 접근 함수를 만든다.
- `npm run validate:data`가 통과하도록 한다.

### 2. 게임 상태

다음 상태를 관리한다.

- 현재 위치
- 현재 날짜: 월/일
- 보유금
- 보유 화물
- 현재 배
- 현재 수레
- 평판
- 스킬
- 선원 상태
- 완료 퀘스트
- 보유 허가장
- 월별 가격표
- 최근 거래/이동 로그

### 3. 모바일 화면

필수 화면은 다음과 같다.

- 시작/불러오기 화면
- 지도 화면
- 항구 상세 화면
- 시장 화면
- 화물/장부 화면
- 배/수레 화면
- 이동 결과/이벤트 화면
- 간단 전투 화면 또는 전투 결과 모달
- 퀘스트/도감 화면

### 4. 이동 시스템

- 현재 위치에서 연결된 루트만 선택할 수 있다.
- 해로는 배가 있어야 한다.
- 육로는 수레/운반 수단이 있어야 한다.
- 이동 시 날짜가 경과한다.
- 루트 위험도에 따라 이벤트가 발생한다.
- 대마도 루트는 `waegwan_pass` 허가장이 없으면 막거나 강하게 경고한다.

### 5. 교역 시스템

- 각 항구의 생산/수요/시장 편향에 따라 구매가와 판매가를 계산한다.
- 월별 이벤트와 상품 계절 보정을 반영한다.
- 화물칸 제한을 적용한다.
- 구매/판매 후 자동 저장한다.
- 거래 후 장부에 최근 손익을 기록한다.

### 6. 월별 가격 갱신

- 매월 1일 또는 월이 바뀔 때 가격표를 갱신한다.
- `monthly_events.json`의 월별 보정값을 반영한다.
- 약간의 랜덤 변동을 추가하되, 저장된 가격표는 같은 달 동안 유지한다.
- 실시간 수요·공급 시뮬레이션은 1차 MVP에서 만들지 않는다.

### 7. 위험 이벤트

- `events.json` 기반으로 이벤트를 표시한다.
- 선택지 효과를 게임 상태에 반영한다.
- 최소 구현 가능 범위:
  - 갯벌 좌초
  - 암초
  - 태풍 조짐
  - 도적대
  - 해적선
  - 관아 검문
  - 순풍

### 8. 간단 전투

초기 전투는 복잡하게 만들지 말고 다음 중 하나로 구현한다.

- 자동 전투 결과 표시
- 또는 `공격 / 회피 / 수리 / 도주` 선택형 전투

전투 결과는 다음에 영향을 준다.

- 선박/수레 내구도
- 화물 손실
- 돈 손실
- 평판
- 선원 사기/피로

### 9. 어업

- 어시장 또는 항구에서 `어업 시도` 버튼을 제공한다.
- 하루를 소모한다.
- 상품 `fresh_fish`, `dried_fish`, `seaweed`, `skate` 중 일부를 획득한다.
- 교역보다 시간 대비 수익이 낮도록 한다.
- 월별 `fishingBonus`를 반영한다.

### 10. 저장

- `localStorage`를 사용한다.
- 자동 저장 조건:
  - 이동 완료
  - 거래 완료
  - 이벤트 완료
  - 월 변경
  - 퀘스트 완료
- 시작 화면에서 새 게임/이어하기를 제공한다.

## 구현 우선순위

1. 데이터 로딩과 타입 안정성
2. 게임 상태와 저장
3. 지도/항구 이동
4. 시장 거래
5. 월별 가격 갱신
6. 이벤트
7. 어업
8. 간단 전투
9. 퀘스트/도감
10. 모바일 UI polish

## 디자인 원칙

- 아이가 이해할 수 있는 짧은 문장과 아이콘 중심 UI를 사용한다.
- 텍스트는 한국어로 표시한다.
- 실패는 존재하되, 완전한 막힘은 피한다.
- 배나 수레가 손실될 수 있지만, 플레이어에게 회복 루트를 제공한다.
- 그래픽은 임시여도 괜찮다. 게임 루프가 우선이다.
- 핵심 UI 요소에는 테스트 가능한 `data-testid`를 붙인다.

## 금지사항

- 데이터를 컴포넌트 내부에 대량 하드코딩하지 말 것.
- 서버/API 의존성을 만들지 말 것.
- 유료·라이선스 불명확 에셋을 포함하지 말 것.
- PC 화면만 고려한 UI로 만들지 말 것.
- 항구 투자, 중국 본토, 일본 본토, 복잡한 실시간 전투를 1차 MVP에 억지로 넣지 말 것.
- UI 개선을 하면서 교역/저장/이동 로직을 임의로 바꾸지 말 것.

## 검증

가능한 범위에서 다음을 실행한다.

```bash
cd starter
npm install
npm run validate:data
npm run build
npm run test:smoke
```

브라우저 검증이 가능하면 `docs/BROWSER_PLAYTEST_PLAN.md`의 필수 플레이 경로를 확인한다.
